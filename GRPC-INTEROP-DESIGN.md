# gRPC / Tonic Interop: Design Analysis

## Should This Be a Separate Project?

No — it should be a feature-gated module in the existing workspace, similar to how `axum-interop` works. But it's a fundamentally different beast from Axum interop.

## Why Axum Interop Was Easy

Axum and typeway share the exact same types — `http::Request<hyper::body::Incoming>`, `http::Response<BoxBody>`, Tower's `Service` trait. The interop is just type conversion between body types. Both frameworks speak the same protocol (HTTP) with the same Rust types.

## Why Tonic Interop Is Hard

gRPC uses Protocol Buffers, not JSON. The request/response types are generated by `protoc`/`prost`, not defined by the user as Rust structs with serde. The service definition lives in `.proto` files, not in Rust types. Tonic's `Service` trait operates on protobuf-encoded bodies, not JSON. So there's a fundamental type mismatch.

## What Interop Could Realistically Look Like

### 1. Shared Tower Middleware (works today, no new code)

Tonic services implement `tower::Service`, so you can apply the same middleware layers (tracing, auth, rate limiting) to both a typeway server and a tonic server in the same binary. No new code needed — just documentation.

### 2. Side-by-Side Serving (works today, needs documentation)

Run both servers on the same port using Tonic's `Routes` multiplexer, which routes HTTP/2 gRPC traffic to Tonic and HTTP/1.1 REST traffic to typeway:

```rust
let rest = typeway_server.into_service();
let grpc = tonic_server.into_service();
// Use hyper to multiplex based on content-type
```

### 3. Shared Types with Dual Serialization (works today)

Derive both `serde` and `prost::Message` on the same types, so a `User` struct works in both REST endpoints and gRPC methods. This is a derive macro concern, not a framework concern.

### 4. API Type → .proto Generation (novel, significant effort)

Generate a `.proto` file from the typeway API type. This requires mapping Rust types to protobuf types and typeway endpoints to gRPC service methods.

This would be a new crate (`typeway-grpc`) in the workspace with a build script or proc macro.

**Mapping challenges:**
- Rust `String` → protobuf `string`
- Rust `u32` → protobuf `uint32`
- Rust `Vec<T>` → protobuf `repeated T`
- Rust `Option<T>` → protobuf `optional T`
- Rust enums (serde-style) → protobuf `oneof` or enums
- Nested structs → nested protobuf messages
- HTTP methods → gRPC unary RPCs (GET → server streaming? POST → unary?)
- Path captures → request message fields

## Recommendation

Items 1-3 work today with no new code — they're patterns to document with examples. Item 4 (API type → proto generation) is genuinely novel but is a significant project — probably a v0.2 feature, not v0.1. It doesn't block anything.
